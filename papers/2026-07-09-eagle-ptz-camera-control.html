Eagle: PTZ 카메라를 직접 제어하는 딥 강화학습

객체 검출-추적-제어를 따로 붙이는 방식 대신, 영상 입력에서 바로 pan/tilt/zoom 동작을 내는 경량 정책을 제안한 PTZ 논문입니다. 조류 충돌 방지 시스템에서는 새를 타깃으로 바꿔 학습하면, PTZ 추적 루프와 엣지 배치의 설계 기준을 바로 얻을 수 있습니다.

요약

Eagle은 이미지에서 객체 위치를 따로 검출한 뒤 제어하는 파이프라인이 아니라, 원시 영상에서 바로 pan/tilt/zoom 동작을 출력하는 엔드투엔드 PTZ 제어 정책이다. 논문의 핵심은 "정확한 박스를 만드는 것"보다 "대상을 계속 화면 중앙 근처에 두는 것"이 실제 추적 성능과 엣지 지연에서 더 중요할 수 있다는 점을 보여준 것이다.
한 줄 판단 새 추적에 그대로 쓰려면 재학습이 필요하지만, PTZ 제어의 구조와 지연 예산을 설계하는 데는 매우 유용하다.
왜 중요하나 Raspberry Pi와 Jetson Nano급 장비에서 실시간으로 돌릴 수 있는 수준의 경량 정책을 제시한다.
현장 연결점 Hanwha Vision PTZ 카메라의 RTSP 입력과 제어 API를 붙일 때, 어떤 부분을 엣지에서 처리해야 하는지 기준이 된다.

왜 이 논문인가

PTZ 중심의 직접 관련성

오늘 주제인 PTZ/active tracking에 정면으로 들어맞는다. 차량이 아니라 새를 추적하더라도, "대상을 FoV 안에 오래 유지하고 화면 중앙에 두는 제어"라는 문제 구조는 거의 같다.

주의할 점

논문은 차량과 사람을 대상으로 학습했다. 조류 시스템에 적용하려면 클래스와 환경을 새로 구성해야 하며, 그대로 가져오면 일반화가 깨질 수 있다.

코드 확인됨

원문 PDF와 GitHub 저장소가 공개되어 있다. EagleSim 시뮬레이터와 학습 코드까지 함께 제공한다.

라이선스 확인됨

실무성 높음

실험에서 30ms 수준의 end-to-end 지연과 엣지 장비 FPS를 제시해, 서버 운영 관점의 판단에 바로 도움이 된다.

핵심 아이디어

핵심 문장

검출기와 트래커를 체인처럼 연결하는 대신, 카메라가 지금 보는 영상만으로 PTZ 행동을 직접 결정한다. 이렇게 하면 중간 단계의 박스 오차와 파이프라인 튜닝 부담을 줄일 수 있다.

논문은 이를 위해 보상 함수와 시뮬레이터를 함께 설계했고, 시뮬레이터에서 학습한 정책을 실세계 영상에도 직접 이식했다.

기술 핵심

보상 함수
중앙 정렬 + 크기 유지 + 잘림 패널티

정답 박스의 중심이 화면 중앙에 가깝고, 대상이 너무 작지도 크지도 않으며, 화면 밖으로 잘리지 않을수록 보상이 커진다. 잘못된 조향을 줄이기 위해 PTZ 수정 자체에도 페널티를 둔다.

학습 설정
120×120 grayscale, 79k params

이미지를 작게 줄이고 회색조로 바꿔 경량화했다. 6개의 병렬 시뮬레이션 환경에서 PPO로 학습했고, 30ms 수준의 센싱-추론-제어 지연을 목표로 했다.

성능 포인트
복잡 장면에서 더 강함

배경이 복잡해질수록 YOLO 기반 파이프라인은 검출 오차에 더 민감해진다. Eagle은 복잡 장면에서 Yolo+Kalman+Controller보다 더 긴 추적 지속 시간을 기록했다.

엣지 지연
Raspberry Pi 4B 33 FPS

논문은 Raspberry Pi 4B와 Jetson Nano에서의 지연을 따로 보고한다. 이 숫자는 서버를 클라우드로 보내지 말고 카메라 근처 엣지에서 처리해야 하는 이유를 분명히 보여준다.

조류 충돌 방지 시스템에 어떻게 쓰나

조류 추적

차량 대신 새를 타깃으로 재학습하면 된다. reward는 "새를 중앙에 오래 유지"로 바꾸고, 데이터는 작은 물체·역광·하늘 배경·군집 비행을 충분히 넣어야 한다.

PTZ 카메라 추적

정답 박스가 불안정할 때는 Eagle처럼 제어 정책이 직접 PTZ를 움직이는 방식이 더 단순하다. 반대로 객체 종류를 자주 바꿔야 하면 다단계 검출 파이프라인이 더 유연할 수 있다.

YOLO 데이터/학습

YOLO를 계속 쓸 경우, 이 논문은 데이터 설계의 방향을 준다. 작은 새, 부분 가림, 배경 나무, 구름, 고정/이동 카메라 높이, 줌 단계까지 학습셋에 넣어야 한다.

실시간 서버 운영

카메라 스트림 처리에서 지연 예산이 가장 중요하다. 30ms 수준을 목표로 하려면 영상 수집, 추론, PTZ 명령 왕복을 같은 엣지 박스 안에서 처리하는 편이 안전하다.

Hanwha Vision / RTSP / PTZ

Hanwha Vision PTZ 카메라의 RTSP 입력을 받아 추론하고, PTZ 제어 API 또는 ONVIF 계층으로 pan/tilt/zoom을 보내는 구조에 잘 맞는다. 이 논문은 그때의 제어 지연과 zoom 과잉을 어떻게 다뤄야 하는지 힌트를 준다.

시뮬레이터 설계

EagleSim의 핵심은 현실적인 배경 변화다. bird collision prevention도 철새, 갈매기, 까마귀 등 서로 다른 크기와 속도를 가진 새를 넣어 domain randomization을 해야 한다.

바로 해볼 실험 3가지

1. 새 추적 직접 정책 vs YOLO+Kalman

같은 RTSP 입력으로 bird detector+Kalman+PTZ 제어와 Eagle 스타일 직접 정책을 비교한다.

측정: 추적 지속 시간, 중심 오차, PTZ 명령 수, 총 지연.

2. 카메라 높이와 줌 민감도

8m, 12m, 20m처럼 설치 높이를 바꿔 보고, 새가 화면 중앙에서 이탈하는 비율을 확인한다.

측정: 거리별 recall, zoom 과잉 빈도, 오검출 후 복원 시간.

3. 배경 혼잡도 랜덤화

하늘, 나무, 해상 풍력 터빈, 역광, 안개, 비를 순차적으로 추가해 정책이 어디서 깨지는지 본다.

측정: 복잡도 증가에 따른 %Tracking 감소폭과 학습 안정성.

용어 정리

용어 뜻
PTZ Pan-Tilt-Zoom. 카메라가 좌우 회전, 상하 회전, 줌을 제어할 수 있는 구조.
FoV Field of View. 카메라가 한 번에 볼 수 있는 시야 범위.
Domain randomization 배경, 조명, 물체 종류를 랜덤하게 바꿔 시뮬레이터-실세계 차이를 줄이는 기법.
Dynamic tasking 배포 중에 추적 대상군을 바꾸는 기능. 이 논문에서는 차량과 사람을 전환한다.
Oracle bounding boxes 시뮬레이터가 제공하는 정답 박스. 검출 오차를 제거한 상한선 비교에 쓴다.
%Tracking 대상을 FoV 안에 유지한 비율. PTZ 추적에서는 가장 직접적인 성능 지표 중 하나다.

  • 새를 중심에 오래 두는 reward를 직접 설계해 본다.
  • RTSP 수신, 추론, PTZ 명령 왕복 시간을 분리해 계측한다.

  • YOLO+Kalman+Controller와 직접 정책을 같은 데이터에서 비교한다.
  • Hanwha Vision 카메라를 실제로 붙일 때는 ONVIF/제어 API와 FPS 제한을 함께 확인한다.
