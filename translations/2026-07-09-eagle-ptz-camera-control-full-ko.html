초록 전문 번역

기존의 PTZ(Pan-Tilt-Zoom) 카메라 자율 제어 방식은 객체 검출과 위치 추정, 그리고 PTZ 메커니즘 제어를 서로 분리된 여러 단계로 수행한다. 이 방식은 수작업 라벨이 필요하고, 단계 사이에서 오차가 누적되며, 객체 검출 신경망의 크기 때문에 자원이 제한된 장비에서 실시간으로 돌리기 어렵다.

우리는 영상 입력을 직접 받아 PTZ 카메라를 제어하는 신경망 정책 Eagle을 학습시키는 종단간(end-to-end) 딥 강화학습(RL) 해법을 제안한다. 현실 세계에서 RL을 학습하는 일은 라벨링 부담, 환경의 불확실성, 실험 장비의 취약성 때문에 매우 번거롭다. 그래서 우리는 PTZ 카메라 제어 정책을 학습하고 평가할 수 있는 사실적인 시뮬레이션 프레임워크 EagleSim을 함께 제안한다.

Eagle은 관심 객체를 캡처된 영상의 중앙 근처에 유지하면서 높은 해상도를 확보하고, 기존 최고 수준 방법보다 최대 17% 더 긴 추적 지속 시간을 보였다. Eagle 정책은 YOLOv5s보다 파라미터 수가 90배 적고, Raspberry Pi와 Jetson Nano 같은 임베디드 카메라 플랫폼에서 각각 33 FPS와 38 FPS로 동작할 수 있어 자원 제약이 있는 환경의 실시간 PTZ 추적에 적합하다. 또한 도메인 랜덤화를 사용하면 시뮬레이터에서 학습한 정책을 실제 환경으로 직접 전이할 수 있다.

논문 본문 전문 번역

1. 서론

능동 시각(active vision)은 시스템이 런타임에 무엇을 볼지 스스로 결정할 수 있게 해준다. PTZ 카메라의 자율 제어는 실시간으로 관심 객체를 추적하면서 이러한 능동 시각을 더 뛰어나게 구현한다. 그래서 원격 감시, 모바일 로보틱스 같은 자원 제약 환경에서 PTZ 카메라는 점점 더 많이 쓰이고 있다.

기존의 자율 PTZ 제어는 대체로 여러 단계로 구성된다. 관심 객체를 검출하고, 그 궤적을 추적한 뒤, PTZ 설정을 제어해 객체를 시야 안에 두는 식이다. 예를 들어 객체 검출에는 신경망을, 궤적 예측에는 칼만 필터를, PTZ 제어에는 PID 제어기를 쓰는 방식이 대표적이다. 하지만 이런 다단계 파이프라인은 각 단계를 따로 튜닝하기 어렵고, 검출기의 오차가 이후 단계로 전파되며, 임베디드 카메라에서 실시간으로 돌리기에도 무겁다.

이런 문제를 피하기 위해 우리는 Eagle이라는 종단간 딥 RL 접근을 제안한다. Eagle은 원시 이미지를 직접 받아 PTZ 동작으로 매핑한다. 즉, 객체 검출, 위치 추정, 제어의 여러 단계를 제거한다. 다만 현실 세계에서 딥 RL을 직접 학습하기는 어렵기 때문에, 우리는 사실적인 가상 세계에서 PTZ 카메라를 배치하고 제어할 수 있는 EagleSim 시뮬레이터도 함께 만든다.

이 논문이 답하려는 질문은 네 가지다. 첫째, 서로 다른 객체를 일반화해서 추적하는 PTZ 제어기를 어떻게 설계할 것인가. 둘째, 여러 제어 단계를 두지 않고 런타임에 추적 목표를 유연하게 바꿀 수 있을까. 셋째, 추적 복잡도가 커질수록 성능은 어떻게 변할까. 넷째, 이런 종단간 PTZ 제어에는 어떤 한계가 있을까.

요약하면, 우리는 (1) 원시 이미지에서 바로 PTZ를 제어하는 경량 Eagle 정책, (2) 사실적인 가상 세계와 PTZ 추상화를 제공하는 EagleSim, (3) 다양한 장면과 객체 변화에 대한 일반화 성능 평가를 제시한다.

2. 배경 및 관련 연구

우리의 목표는 PTZ 카메라의 시야(Field of View, FoV) 안에 관심 객체를 높은 해상도로 유지하는 것이다. 자율 제어기는 pan과 tilt를 조정해 객체를 영상 중앙에 두고, zoom을 조절해 객체가 잘리지 않으면서도 가능한 크게 보이게 해야 한다. 객체를 중앙에 두면 급격한 방향 전환이 일어났을 때 시야 밖으로 빠져나갈 위험을 줄일 수 있다.

원문 Figure 2는 PTZ 제어를 네 가지 방식으로 나눠 비교한다. 객체 검출+추적+제어, 객체 검출+RL, 상대 위치+제어, 그리고 Eagle의 종단간 deep-RL이다. 기존 다단계 방식은 검출기, 추적기, 제어기 사이의 연결이 복잡하고, 검출기 오류와 라벨링 비용에 취약하다. 반면 Eagle은 영상에서 바로 행동을 내기 때문에 파이프라인의 중간 손실을 줄일 수 있다.

특히 신경망 기반 객체 검출기를 쓰는 방식은 유연하지만, 모바일이나 임베디드 PTZ 카메라에서 돌아가기에는 무겁다. 또한 상대 위치를 예측하는 방식은 학습된 위치 예측기와 별도 제어기를 다시 맞춰야 하고, 일반화가 쉽게 깨진다. 따라서 PTZ 제어를 하나의 정책으로 통합하는 접근이 필요하다.

3. Eagle의 설계

3.1 단일 객체 추적을 위한 보상 함수

우선 우리는 장면에 다른 객체가 없다고 가정한 상태에서 단일 관심 객체를 추적하는 보상 함수를 정의한다. 이때 목적은 관심 객체를 화면 중심에 가깝게 두면서 최대한 높은 해상도로 유지하는 것이다. 객체의 bounding box 중심 좌표와 영상 크기를 이용해, 중심 정렬 정도와 객체 크기, 그리고 화면 잘림(clipping) 여부를 보상에 반영한다.

객체가 영상 안에 존재하면 보상은 중심 정렬과 객체 크기, 잘림 여부를 곱한 값에서 PTZ 변경 페널티를 뺀 형태가 된다. 객체가 캡처되지 않으면 에이전트는 음의 보상 L을 받는다. 이 페널티는 카메라를 너무 자주 흔들지 않도록 만든다.

3.2 일반화 가능한 PTZ 추적

다음 질문은, 서로 다른 종류의 객체를 추적해야 할 때도 같은 정책이 잘 동작하도록 만들 수 있느냐는 것이다. 우리는 장면 안의 객체를 두 클래스로 나눈다. 하나는 추적 대상 집합 A이고, 다른 하나는 무시해야 할 집합 B다. 예를 들어 차량 추적 장면에서는 A에 여러 차종과 색상이 포함되고, B에는 배경 건물, 나무, 사람 같은 객체가 들어간다.

핵심은 동일한 정책이 A에 속한 여러 객체에 일반화되도록 학습하는 것이다. 한 시점에는 A의 객체가 하나만 있다고 가정하고, 정책은 그 객체를 추적하도록 보상을 받는다. 이렇게 하면 차량 종류나 색상이 바뀌어도 하나의 정책으로 대응할 수 있다. 새 추적 시스템으로 옮기면, 이 A 집합을 새 종 또는 새의 하위 유형으로 바꾸면 된다.

3.3 동적 태스킹: 유연한 추적 목표

우리는 런타임에 추적 목표를 바꾸는 동적 태스킹(dynamic tasking)도 제안한다. 기존 다단계 방식은 객체 검출 단계의 필터링만 바꾸면 추적 목표를 쉽게 바꿀 수 있지만, Eagle에는 별도의 객체 검출기가 없다. 그래서 우리는 현재 카메라 이미지와 함께 추가적인 문맥 입력(contextual input)을 넣고, 보상 계산을 바꿔서 추적 목표를 전환한다.

예를 들어 차량과 사람 두 하위 클래스를 두고, 문맥 입력 CI가 0이면 차량 집합을, 1이면 사람 집합을 추적하도록 할 수 있다. 이 방식은 특정 문맥과 특정 객체군을 연결하도록 에이전트가 배우게 만든다. 다만 목표를 유연하게 바꾸는 대신 성능 비용이 조금 생길 수 있다.

4. EagleSim의 설계

4.1 Eagle를 학습시키는 방법

우리는 Eagle을 학습시키는 세 가지 방법을 검토한다. 첫째, 특수한 영상만 써서 학습하는 방법이다. 하지만 PTZ 카메라는 pan/tilt/zoom을 바꾸면 시야가 바뀌므로, 단일 시점의 일반 영상만으로는 부족하다. 둘째, 현실 세계에서 바로 학습하는 방법이다. 그러나 데이터 라벨링, 실험 환경의 불안정성, 다양한 장면 구성의 어려움 때문에 사실상 매우 어렵다. 셋째, 시뮬레이터에서 학습하는 방법이다. 우리는 이 방식을 택하고, 필요하면 현실 세계로 미세조정(finetuning)하는 하이브리드 접근도 가능하다고 본다.

4.2 사실적인 가상 세계

EagleSim은 Unreal Engine과 AirSim을 바탕으로 만들어졌다. AirSim은 드론과 차량 제어는 지원하지만, 서로 다른 차량 종류, 사람 캐릭터, 배경 나무, 경계, 주변 환경의 세밀한 변화를 충분히 다루지 못한다. 우리는 이런 문제를 해결하기 위해 PTZ 소프트웨어 추상화를 포함한 커스텀 가상 세계를 만들었고, 차량 추적과 사람 추적 시나리오를 패키지로 제공한다.

가상 세계는 차량 6종과 색상 5종, 사람 캐릭터 6종, 배경 재질/패턴 25종, 나무 10종을 지원한다. 또한 그림자, salt-and-pepper 노이즈, 무작위 대비, 밝기 변화 같은 영상 증강도 적용할 수 있다. 각 세계는 70m x 70m의 열린 공간이며, 경계를 숨기면 단순한 장면이 되고, 배경 패턴과 나무를 넣으면 도시, 숲, 농촌 같은 복잡한 장면이 된다.

이처럼 다양한 장면 변형을 넣는 이유는 simulation-to-reality gap을 줄이기 위해서다. 실제 영상에는 배경, 그림자, 가림, 조명처럼 시뮬레이터에서 보지 못한 변화가 많다. 그래서 우리는 도메인 랜덤화를 사용해 여러 장면 변형이 섞인 정책을 학습한다.

4.3 PTZ 추상화

EagleSim의 PTZ 추상화는 네 개의 모듈로 이뤄진다. 카메라 배치 모듈, 카메라 컨트롤러, 오브젝트 트래커, 장면 컨트롤러다. 카메라는 1도 단위로 pan과 tilt를 조절할 수 있고, zoom은 horizontal FoV를 1도 단위로 바꾸는 방식으로 구현된다. vertical FoV는 영상의 종횡비로부터 계산된다.

오브젝트 트래커는 Unreal Engine의 3D 외곽 메쉬를 2D 이미지 좌표로 변환해 정답 bounding box를 제공한다. 이 bounding box는 오차가 거의 없는 정답 상자이기 때문에, 학습 보상을 계산하거나 다단계 파이프라인의 상한선을 비교할 때 유용하다. 원문 Figure 3는 이런 구성 요소가 Eagle 정책과 어떻게 연결되는지 보여준다.

5. 평가

5.1 PTZ 추적 성능 지표

Eagle은 중간 bounding box를 생성하지 않기 때문에, 다단계 방법에서 흔히 쓰는 지표를 그대로 적용할 수 없다. 그래서 우리는 %Tracking, Center_x, Center_y, Obj_size 같은 지표를 직접 사용해 카메라 제어 성능을 평가한다. %Tracking은 대상 객체를 카메라 FoV 안에 유지한 시간 비율이며, 100%면 전체 에피소드 동안 놓치지 않았다는 뜻이다.

Center_x와 Center_y는 영상 중앙에 대상이 얼마나 잘 유지됐는지 나타내고, Obj_size는 영상 안에서 객체가 차지하는 상대 크기를 나타낸다. PTZ 제어의 목표는 대상이 화면 중앙에 머무르면서 충분히 크게 보이도록 하는 것이므로, 이 세 지표가 함께 중요하다.

5.2 Eagle의 구현

학습 시 입력 영상은 120x120으로 줄이고 grayscale로 변환한다. 큰 컬러 이미지를 직접 쓰면 계산 비용이 크기 때문이다. 정책 네트워크는 7개의 은닉층과 출력층으로 구성되며, Atari 게임용 네트워크 구조에서 영감을 받았다. 처음 다섯 층은 convolution layer이고, 마지막 세 층은 fully-connected layer다. 출력은 pan, tilt, zoom을 제어하는 3개의 이산 동작이다.

네트워크는 총 79k 파라미터로 매우 가볍다. 6개의 병렬 시뮬레이션 환경을 사용해 하나의 정책을 69시간 동안 학습했고, PPO(Proximal Policy Optimization) 알고리즘을 사용했다. 각 에피소드는 2000 step이며, 이는 약 1분의 연속 추적에 해당한다.

5.3 추적 시나리오

우리는 추적 복잡도가 점점 커지는 다섯 개의 차량 추적 시나리오 Sc-1부터 Sc-5와, 차량 또는 사람을 바꿔 추적하는 dynamic tasking(DT) 시나리오를 정의한다. Sc-1은 고정 배경에서 단일 SUV를 추적하는 가장 쉬운 장면이고, Sc-5는 배경, 나무, 사람, 이미지 증강이 모두 들어간 가장 복잡한 장면이다.

학습 중 차량은 70m x 70m의 열린 공간에서 무작위 경로를 따른다. 평균 속도는 6m/s, 최대 속도는 16m/s다. PTZ 카메라는 남쪽 경계의 중간 지점, 높이 8m에 설치되고, 처음에는 넓은 장면을 본 뒤 차량이 움직이면 이를 따라가면서 zoom을 조절한다. 원문 Figure 4는 이런 시나리오와 학습 보상 곡선을 보여준다.

5.4 Eagle과 기존 방법의 비교

우리는 기존의 세 가지 계열과 비교한다. 첫째, YOLOv5s를 객체 검출기로 쓰고 Kalman filter와 rule-based controller를 붙인 Yolo+Kalman+Controller. 둘째, bounding box를 입력으로 RL 정책을 학습하는 Yolo+Deep_RL. 셋째, 상대 위치를 예측하는 NN+Controller. 공정 비교를 위해 모든 방법의 센싱에서 PTZ 적용까지의 end-to-end 지연은 30ms로 맞췄다.

결과적으로 Eagle은 고정 배경에서 Yolo+Kalman+Controller보다 4.6% 더 긴 추적을 보였고, 복잡한 장면(Variable backgrounds+Trees+Humans)에서는 23.8% 더 좋은 추적 지속 시간을 기록했다. 완전한 bounding box를 쓸 수 있는 상한선 설정과 비교해도 Eagle은 추적 지속 시간 면에서 근접하거나 그 이상을 보였다. 즉, 원시 영상이 박스 정보보다 더 풍부한 방향성 정보를 담을 수 있다는 해석이 가능하다.

5.5 경량 객체 검출기 비교

우리는 EagleSim이 제공하는 정답 박스를 이용해 아주 가벼운 커스텀 객체 검출기도 학습했다. 50k, 100k, 150k 장의 데이터셋을 만들고, 그 결과를 Kalman filter와 rule-based controller와 결합해 PTZ 추적기를 구성했다. 데이터셋이 커질수록 성능이 조금씩 좋아지지만, Eagle은 이런 다단계 파이프라인보다 여전히 우수했다.

5.6 실제 영상으로의 전이

우리는 시뮬레이터에서 학습한 Eagle 정책을 실제 장면 영상에도 직접 적용해 보았다. 초기에는 넓은 시야에서 대상이 들어있는 bounding box를 손으로 선택한 뒤, 정책이 pan과 tilt, zoom을 어떻게 바꾸는지 시뮬레이션한다. 단순한 장난감 SUV 장면과 나무/패턴이 있는 실제 차량 장면 둘 모두에서 Eagle은 대상 차량을 화면 중앙에 유지했다.

원문 Figure 7은 이런 전이 결과를 보여준다. EagleSim에서 학습한 정책이 실제 영상에 바로 적용될 수 있다는 점은 sim2real 전이의 가능성을 보여주는 중요한 증거다.

5.7 엣지 카메라에서의 실행 시간

우리는 Raspberry Pi 4B와 Jetson Nano에서 Eagle 정책의 실행 시간을 측정했다. Eagle은 79k 파라미터만 사용하며, Raspberry Pi에서는 약 9.2ms, Jetson Nano에서는 약 6.2ms의 추론 지연을 기록했다. 여기에 센싱과 actuation 지연을 더하면 Raspberry Pi에서는 약 30ms 정도의 end-to-end 지연으로 33 FPS 실시간 배치가 가능하다.

같은 환경에서 최적화한 Yolov5s는 훨씬 느리다. Raspberry Pi에서는 실시간 배치가 사실상 어렵고, Jetson Nano에서도 end-to-end 지연이 훨씬 커진다. 따라서 자원 제약이 있는 PTZ 카메라에서는 경량 정책이 강한 장점이 있다.

6. 논의

여러 개의 관심 객체가 동시에 있는 장면에서는 Eagle 정책이 더 큰 객체, 그리고 더 높은 기대 보상을 주는 객체를 추적하는 경향을 보인다. 이는 보상 함수가 화면 중앙에 더 크게 보이는 객체를 선호하기 때문이다. Figure 8은 이런 다중 객체 장면의 예시를 보여준다.

객체 속도도 중요한 변수다. 이 논문에서 평가한 차량은 평균 6m/s, 최대 16m/s 정도로 움직였다. 더 빠른 객체를 추적하려면 action space를 더 세분화하거나, PTZ 변경 지연 자체를 줄여야 한다. 사람 추적도 가능하지만, 복잡도 차이 때문에 차량보다 성능이 달라질 수 있다.

다중 카메라 협동 추적도 가능하지만, 이 논문은 그 부분을 미래 과제로 남긴다. EagleSim은 여러 PTZ 카메라를 같은 장면에 둘 수 있지만, 카메라 간 협업을 위한 종단간 제어 정책은 별도로 연구해야 한다. 또한 새로운 객체 유형으로 목표를 바꾸려면 재학습이 필요하다는 한계도 있다.

즉, Eagle은 범용 객체 검출기처럼 언제든 목표 클래스를 바꾸는 데 최적화된 시스템은 아니지만, 이미 어떤 대상군을 추적할지 알고 있고, 가볍고 빠른 제어가 필요한 상황에서는 매우 강력하다.

7. 결론

우리는 원시 영상을 직접 입력받아 PTZ 카메라를 제어하는 경량 종단간 접근 Eagle을 제안했다. Eagle은 다양한 추적 복잡도 시나리오에서 기존 방법보다 뛰어난 성능을 보였고, 보상 함수와 시뮬레이터 설계를 통해 일반화 가능한 정책을 학습할 수 있음을 보여주었다.

또한 EagleSim을 통해 PTZ 카메라 제어기의 학습과 평가를 자동화할 수 있었고, 시뮬레이터에서만 학습한 정책이 실제 영상으로도 직접 전이될 수 있음을 보였다. 이는 현재의 다단계 PTZ 접근을 대체할 수 있는 유망한 방향이다.

수식과 용어

r_i^a = { Center_x^a × Center_y^a × Obj_size^a × Clip^a - P, if Condition; -L, otherwise }

식 (1)은 단일 객체 추적에서의 한 step 보상이다. 객체가 보이는 경우에는 중심 정렬, 객체 크기, 잘림 여부를 곱한 뒤 PTZ 변화 페널티 P를 뺀다. 객체가 보이지 않으면 음의 보상 L을 준다.

Center_x^a = abs(Width/2 - x) / (Width/2)
Center_y^a = abs(Height/2 - y) / (Height/2)
Obj_size^a = ((Xmax - Xmin) × (Ymax - Ymin)) / (Width × Height)

식 (2)는 중심 오차와 상대 객체 크기를 정의한다. 값이 작을수록 중앙에 가깝거나 객체 크기가 작다. 이 논문에서는 두 중심 오차와 객체 크기 값을 모두 [0, 1] 구간으로 정규화한다.

Clip^a = M if Xmin = -(Height/2) or Ymin = -(Width/2)
Clip^a = M if Xmax = (Height/2) or Ymax = (Width/2)
Clip^a = 1 otherwise

식 (3)은 객체가 영상의 경계에서 잘릴 때 보상을 줄이는 항이다. M은 0과 1 사이의 하이퍼파라미터다.

R^a = Σ_{i=0}^{N} r_i^a

식 (4)는 한 에피소드 전체의 총 보상이다. step 보상을 모두 더해 정책의 학습 신호로 사용한다.

Condition = True if C_v or C_h; False otherwise

식 (5)는 dynamic tasking에서의 조건이다. C_v는 차량 하위 클래스와 문맥 입력이 맞을 때, C_h는 사람 하위 클래스와 문맥 입력이 맞을 때 참이 된다.

용어 설명
PTZ Pan-Tilt-Zoom. 카메라의 좌우/상하 회전과 줌을 뜻한다.
FoV Field of View. 카메라가 한 번에 볼 수 있는 시야다.
Domain randomization 배경, 조명, 객체 모양을 랜덤하게 바꿔 시뮬레이터와 현실의 차이를 줄이는 기법이다.
Dynamic tasking 런타임에 추적 목표를 바꾸는 기능이다.
Oracle bounding boxes 시뮬레이터가 주는 정답 박스다. 검출 오차가 없는 상한선 비교에 쓴다.
%Tracking 대상을 FoV 안에 유지한 시간 비율이다.

번역자 메모

  • active vision은 능동 시각으로 번역했다. 단순한 추적이 아니라 "어디를 볼지 스스로 결정하는" 의미를 살리기 위해서다.
  • dynamic tasking은 배포 중 추적 목표를 바꾸는 기능으로 옮겼다. 원문의 의도를 가장 잘 드러내는 표현이다.
  • oracle bounding boxes는 정답 박스로 옮겼다. 시뮬레이터가 제공하는 오차 없는 상자라는 뜻이다.
  • FoV는 시야로 번역했다. PTZ 문맥에서는 카메라가 현재 덮는 영역을 의미한다.
  • 표의 수치와 단위는 원문 표기와 맞추기 위해 영문 약어를 남겼다. 실제 재현 시에는 원문 PDF와 함께 확인하는 것이 안전하다.

실무 읽기 노트

bird detection

이 논문은 새를 직접 다루지 않지만, 작은 객체를 놓치지 않고 중심에 유지하는 보상 설계가 핵심이다. 새 추적용으로 바꾸려면 클래스 A를 새 종으로 바꾸고, 비행 방향과 가림을 더 강하게 넣어야 한다.

PTZ camera tracking

정확한 bounding box보다 추적 지속 시간과 중심 오차가 더 중요할 수 있다는 점을 보여준다. PTZ 제어 루프를 짤 때는 검출, 추적, 제어를 한 장비에 넣을지 먼저 계산해야 한다.

RTSP / edge / server

30ms 수준의 end-to-end 지연이 목표라면, 서버 왕복보다 엣지 처리에 더 큰 비중을 둬야 한다. RTSP ingest, 추론, PTZ 명령 발행이 한 박스 안에서 끝나야 한다.

YOLO / training data

이 논문은 YOLO를 버리라는 뜻이 아니라, YOLO가 병목이 될 수 있음을 보여준다. 새 데이터셋을 만든다면 작은 새, 부분 가림, 하늘-나무 배경, 다양한 고도와 줌을 포함해야 한다.

ONNX / TensorRT / TFLite

논문은 TFLite와 TensorRT로 지연을 줄였다. 실제 제품에서는 ONNX export 후 TensorRT 또는 OpenVINO, TFLite 중 어떤 경로가 배포 장비와 맞는지 먼저 확인하면 된다.

Hanwha Vision 연동

Hanwha Vision PTZ 카메라를 쓴다면 RTSP 스트림을 받아 추론하고, 카메라 제어 API 또는 ONVIF로 pan/tilt/zoom을 전달하는 구조가 자연스럽다. 이 논문은 그런 루프를 얼마나 가볍게 만들어야 하는지 기준을 준다.

실무 주의

이 논문은 차량과 사람으로 검증되었다. 조류 시스템에 넣을 때는 객체 크기, 운동 패턴, 가림, 배경 복잡도를 다시 맞춰야 한다. 그대로 복사하면 성능이 흔들릴 가능성이 크다.
